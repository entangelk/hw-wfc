# HW-WFC: 제약 기반 붕괴적 탐색을 활용한 AI 하드웨어 자동 스케줄링

## 1. 알고리즘 개요

### 1.1 HW-WFC란?

2D 경로 탐색을 위해 고안된 **Circle-WFC의 핵심 사상**(플러그인 제약, 중간 지점 양방향 붕괴)을 **다차원 하드웨어 최적화 공간**에 적용한 AI 컴파일러 스케줄링 알고리즘이다.

수십억 개에 달하는 하드웨어 파라미터 조합을 전부 탐색하는 대신, **물리적 제약 조건을 통해 불가능한 상태를 연쇄적으로 붕괴(Collapse)시켜** 최적의 메모리 레이아웃과 연산 구조를 찾아낸다.

### 1.2 핵심 아이디어 매핑

| Circle-WFC (2D 경로 탐색) | HW-WFC (하드웨어 최적화) | 본질 |
|---|---|---|
| Grid / Tile Map | 하드웨어 탐색 공간 (Search Space) | 타일 크기, 루프 전개, 메모리 레이아웃 |
| 원주(Circle) 제약 | 물리적 하드웨어 한계선 | SRAM 용량, 메모리 대역폭, 레지스터 한계 |
| 중간 교점 시작 (양방향 붕괴) | 병목 레이어 중심 선행 붕괴 | 가장 무거운 연산부터 고정, 앞뒤로 전파 |
| 플러그인 조건 주입 | 타겟 하드웨어 스펙 주입 (JSON) | 하드웨어가 바뀌면 스펙만 교체 |

## 2. 핵심 메커니즘

### 2.1 하드웨어 상태의 중첩 (Superposition)

각 레이어(노드)는 컴파일 전, 여러 하드웨어 구현 방식이 중첩된 상태로 존재한다:

- **Memory Layout**: `Row-major`, `Col-major`, `Block-tiled`
- **Tiling Size**: `16×16`, `32×32`, `64×64`, `128×128`, 직사각 타일 등
- **Compute Location**: `SRAM (Shared Memory)`, `DRAM (Global Memory)`

### 2.2 2단계 제약 엔진

단순한 '가능/불가능'을 넘어, 하드웨어의 성능적 트레이드오프를 판단하는 2단계 시스템:

**Phase 1 — Hard Constraints (물리적 불가능 소거)**
- SRAM/L1 캐시 용량 초과, 메모리 정렬(Alignment) 위반 등 물리적으로 실행 불가능한 상태를 즉시 삭제
- 목적: 탐색 공간의 폭발적 증가를 초기에 차단

**Phase 2 — Soft Constraints & Cost Model (확률적 랭킹)**
- Roofline 모델 기반 경량 비용 모델로 각 상태의 예상 성능 점수를 계산
- 점수를 정규화하여 확률로 변환 → Shannon 엔트로피 계산 → 가장 확실한 상태부터 붕괴

### 2.3 병목 우선 양방향 붕괴

일반적인 컴파일러처럼 입력→출력 순차 탐색하지 않는다:

1. **병목 탐색**: 전체 모델에서 FLOPs가 가장 높은 레이어를 시작점으로 설정
2. **선행 붕괴**: Cost Model 점수가 가장 높은 상태로 강제 붕괴
3. **양방향 전파**: 확정된 제약이 이전/이후 레이어로 파동처럼 전파
4. **연쇄 붕괴**: 주변 레이어가 전파된 제약에 맞춰 자동 붕괴

### 2.4 백트래킹

선행 붕괴가 전역 최적을 깨트리는 것을 방지하는 안전장치:
- 전파 과정에서 모순 발생(선택 가능한 상태 0개) → 가장 최근 붕괴를 취소
- 해당 상태를 후보에서 제외하고, 차순위 상태로 재붕괴

### 2.5 전환 비용 통합

인접 레이어 간 설정이 다르면 데이터 변환 비용이 발생한다:
- Layout 전환: ROW↔COL (전치), ROW↔BLOCK (부분 재배열)
- 위치 전환: SRAM↔DRAM (메모리 계층 이동)
- 타일 크기 전환: 면적 비율에 비례하는 re-tiling 비용

붕괴 시 인접 노드와의 전환 비용을 base score에서 차감하여, **단독 최적해보다 파이프라인 전체 최적해를 선호**하도록 한다. 가중치는 SRAM 용량에서 자동 파생된다.

## 3. 기존 방식과의 비교

| 항목 | Grid Search / RL | HW-WFC |
|------|-----------------|--------|
| **탐색 방식** | 무작위 변형 후 실행 측정 (Trial & Error) | 제약 조건에 의한 불가능 상태 소거 (Deductive) |
| **시간 복잡도** | 수천~수만 번의 컴파일 필요 | 연쇄 붕괴로 경우의 수 급감 (~2-3ms) |
| **최적화 방향** | 순방향 (입력→출력) | 양방향 (병목→양쪽 동시 전파) |
| **하드웨어 변경** | 처음부터 재탐색 | 스펙(JSON) 교체만으로 즉시 적용 |
| **레이어 간 상호작용** | 독립 최적화 (상호작용 미반영) | 전환 비용 + 전파로 전역 최적화 |

## 4. 검증 전략

### 4.1 Toy Model (기본 검증)

**구조**: `Linear(A)` → `ReLU(B)` → `Linear(C)` / SRAM 64KB

제약 전파만으로 탐색 공간이 98.5% 감소하는지 확인하는 최소 검증.

### 4.2 Transformer Attention Block (핵심 검증)

**구조**: `QKV Proj` → `QK MatMul` → `Softmax` → `AV MatMul` → `Out Proj` → `LayerNorm`

검증 포인트:
- Softmax는 메모리 병목 레이어. MatMul이 선택한 레이아웃과 충돌할 때 HW-WFC가 감지하는가?
- 전환 비용 전파를 통해 "MatMul 단독 최적을 포기하고 전체 Attention Block의 속도를 높이는 전역 최적"을 찾는가?
- SRAM이 tight(12KB)할 때 레이어 타입별 다른 최적 타일을 발견하는가? (복사기 문제 해결)

### 4.3 Safety Checks (5가지 안전 점검)

"결과가 좋아 보이는데, 진짜 좋은 건가?"를 의심하는 연구 원칙:

1. **Hard Constraint 정확성** — 유효한 후보를 잘못 제거하지 않는가?
2. **Cost Model 변별력** — 점수가 flat이면 엔트로피 선택 ≈ 랜덤
3. **전파 실효성** — 전파가 명목상이 아니라 실질적 효과를 내는가?
4. **백트래킹 활성화** — dead code가 아니라 실제 발동하는가?
5. **복사기 문제 없음** — 이질적 레이어가 다른 최적 상태를 선택하는가?
