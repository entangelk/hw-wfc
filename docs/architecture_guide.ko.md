# HW-WFC 아키텍처 해설서

> 이 문서는 프로젝트를 처음 접하는 사람이 **"왜 이렇게 설계했는가"**를 이해하기 위한 해설서다.
> 코드 구조나 API는 다루지 않는다. 그것은 코드를 읽으면 알 수 있다.
> 여기서는 코드만 봐서는 알 수 없는 **의도, 배경, 트레이드오프**를 설명한다.

---

## 1. 이 프로젝트가 풀려는 문제

AI 모델을 GPU에서 실행할 때, 각 레이어(MatMul, Softmax, ReLU 등)마다 결정해야 할 것이 많다:

- **타일 크기**: 행렬을 몇×몇 조각으로 나눠서 처리할 것인가? (8×8, 32×32, 64×128...)
- **메모리 레이아웃**: 데이터를 행 우선(Row-Major)으로 배치할 것인가, 열 우선(Col-Major)으로 할 것인가?
- **연산 위치**: 빠르지만 작은 SRAM에서 할 것인가, 느리지만 큰 DRAM에서 할 것인가?

이 조합이 레이어당 수십~수백 개이고, 레이어가 수십 개이면 전체 탐색 공간은 **수십억 조합**이 된다.

### 기존 방식의 한계

| 방식 | 문제 |
|------|------|
| Grid Search | 모든 조합을 시도. 탐색 공간이 기하급수적으로 폭발 |
| RL 기반 탐색 | 학습에 수천~수만 번의 실제 실행이 필요. 하드웨어가 바뀌면 처음부터 다시 |
| 수동 튜닝 | 전문가 의존. 레이어 간 상호작용을 놓치기 쉬움 |

### HW-WFC의 접근

**"가능한 조합을 전부 시도하지 말고, 불가능한 것을 먼저 제거하자."**

Wave Function Collapse(WFC) 알고리즘의 핵심 아이디어를 차용한다:
1. 모든 레이어가 모든 가능한 상태를 동시에 갖는다 (중첩, Superposition)
2. 물리적으로 불가능한 상태를 제거한다 (Hard Constraint)
3. 가장 확실한 레이어부터 하나의 상태로 확정한다 (붕괴, Collapse)
4. 확정된 결과가 이웃 레이어에 파급된다 (전파, Propagation)
5. 막다른 길에 도달하면 되돌린다 (백트래킹, Backtracking)

## 2. GPU 스펙 설계 의도

### 2.1 왜 가상 스펙인가?

실제 A100/H100 스펙을 쓰지 않은 이유:

- **실제 GPU의 SRAM(192~256KB)은 너무 넉넉하다.** 우리의 테스트 타일(최대 128×128 = 64KB)이 전부 SRAM에 들어가 버려서, 제약이 발동하지 않고 알고리즘이 검증되지 않는다.
- **알고리즘이 "쉬운 문제"만 풀고 있다는 착각을 방지해야 한다.** SRAM이 넉넉하면 모든 레이어가 같은 최적 타일을 고르게 되고, 전파도 백트래킹도 불필요해진다. 이것은 알고리즘이 잘 되는 것이 아니라, 문제가 쉬운 것이다.

### 2.2 세 가지 스펙의 역할

```
스펙 설계 = "알고리즘의 어떤 측면을 시험하고 싶은가?"에서 출발
```

| 스펙 | SRAM | Alignment | 역할 |
|------|------|-----------|------|
| `toy_gpu` | 64KB | 128B | **기본 동작 확인**. 충분한 SRAM에서 알고리즘이 올바른 최적해를 찾는지 |
| `tight_gpu` | 16KB | 64B | **전이 지점 탐색**. 일부 타일이 SRAM에 못 들어가기 시작 |
| `stress_gpu` | 12KB | 64B | **핵심 검증 스펙**. SRAM이 극도로 부족하여 레이어별 다른 최적해가 존재해야 함 |

### 2.3 SRAM 크기별 의도를 구체적으로

#### toy_gpu (64KB)

```
32×32 float32 타일 = 4KB   → SRAM의 6%만 사용. 여유로움
64×64 float32 타일 = 16KB  → SRAM의 25%. 아직 여유
128×128 float32 타일 = 64KB → SRAM의 100%. 경계
```

- 대부분의 타일이 SRAM에 들어간다.
- 목적: **알고리즘의 기본 흐름**(붕괴→전파→완료)이 정상 동작하는지 확인.
- 한계: 모든 레이어가 같은 최적 타일(64×64)을 고른다 → 차별화 없음.

#### stress_gpu (12KB) — 가장 중요한 스펙

```
32×32 float32 타일 = 4KB
  → LINEAR(working memory 3배) = 12KB → SRAM 100%. 빈틈없이 딱 맞음
  → SOFTMAX(working memory 4배) = 16KB → SRAM 초과! 사용 불가

16×16 float32 타일 = 1KB
  → SOFTMAX(4배) = 4KB → SRAM의 33%. 넉넉
```

이것이 의미하는 것:
- **LINEAR 레이어는 32×32가 최적** (SRAM을 꽉 채워서 효율 극대화)
- **SOFTMAX 레이어는 16×16이 최적** (32×32를 쓰면 SRAM 초과)
- 같은 모델 안에서 **레이어마다 다른 최적 타일이 존재**한다
- 이것을 알고리즘이 스스로 발견하는지가 핵심 검증 포인트 ("복사기 문제" 해결 여부)

#### working memory 배수란?

타일 하나를 처리할 때 SRAM에 동시에 올려야 하는 버퍼 수:

| 레이어 | 버퍼 | 배수 |
|--------|------|------|
| LINEAR (MatMul) | input_tile + weight_tile + output_tile | 3× |
| SOFTMAX | input + exp_buffer + sum_buffer + output | 4× |
| RELU | input + output | 2× |

예: 32×32 타일(4KB)로 LINEAR를 처리하면 SRAM에 4KB × 3 = 12KB가 동시에 필요하다.
stress_gpu(12KB)에서는 이것이 SRAM의 100%를 차지한다.

### 2.4 Alignment 설정 의도

```
alignment_bytes = 128  (toy_gpu)
```

GPU의 메모리 컨트롤러는 특정 바이트 경계에 정렬된 주소에서만 효율적으로 데이터를 읽는다.
128B alignment는 실제 A100의 값을 참고했다.

이것이 실제로 하는 일:
- tile_n × dtype_bytes가 alignment의 배수여야 한다
- 8×8 타일: row = 8 × 4 = 32B → 32 % 128 ≠ 0 → **제거됨**
- 16×16 타일: row = 16 × 4 = 64B → 64 % 128 ≠ 0 → **제거됨**
- 32×32 타일: row = 32 × 4 = 128B → 128 % 128 = 0 → **통과**

이것은 의도된 동작이다. 실제 GPU에서도 작은 타일은 메모리 대역폭을 낭비한다.

### 2.5 나머지 숫자들

| 파라미터 | 값 | 출처 |
|----------|-----|------|
| `compute_gflops: 312000` | NVIDIA A100 FP32 peak | 공식 스펙시트 |
| `sram_bandwidth_gbps: 19000` | A100 Shared Memory 이론 대역폭 | 아키텍처 문서 |
| `dram_bandwidth_gbps: 900` | A100 HBM2e | 공식 스펙시트 |

이 값들은 Roofline 모델에서 "연산 병목 vs 메모리 병목"을 구분하는 데 사용된다.
SRAM 크기만 가상이고, 나머지 성능 수치는 실제 GPU 기반이다.

## 3. 비용 모델(Cost Model)이 점수를 매기는 방법

각 후보 상태(HWState)에 점수를 부여한다. 점수 = "이 타일 구성이 얼마나 효율적인가?"

### 3.1 세 가지 점수 요소

```python
총점 = roofline_score + layout_affinity + cache_efficiency
```

#### Roofline Score (0~1.0)

"이 연산이 연산 병목인가, 메모리 병목인가?"를 판단한다.

```
Operational Intensity = FLOPs / Memory Traffic
```

- OI가 높으면 → 연산 병목 → GPU 코어를 충분히 활용 → 높은 점수
- OI가 낮으면 → 메모리 병목 → GPU가 데이터를 기다림 → 낮은 점수

타일이 크면 데이터 재사용률이 높아져 Traffic이 줄고 OI가 올라간다.
이것이 "큰 타일 = 좋다"의 근거이지만, SRAM 용량 제약과 트레이드오프된다.

#### Layout Affinity (0~0.2)

레이어 타입에 따라 선호하는 메모리 레이아웃이 다르다:

- **Softmax**: 행 단위 reduction → ROW_MAJOR 필수 (+0.2)
- **LINEAR**: 출력 write에 ROW_MAJOR 유리 (+0.15)
- **Conv2D**: 공간적 지역성에 BLOCK_TILED 유리 (+0.15)
- **RELU**: element-wise라 레이아웃 무관 (+0.05)

#### Cache Efficiency (-0.3~+0.2)

SRAM 사용률이 최적 지점(75%)에 가까울수록 높은 보너스.

```
너무 적게 사용 (20%) → SRAM 낭비 → 낮은 점수
딱 맞게 사용 (75%) → 이상적 → 최고 점수
초과 사용 (>100%) → 물리적 불가 → 패널티
```

가우시안 곡선으로 모델링하여, 0.75 주변에서 피크를 가진다.

### 3.2 왜 이 세 요소인가?

| 요소 | 포착하는 현상 | 없으면 생기는 문제 |
|------|-------------|-------------------|
| Roofline | 타일 크기에 따른 연산/메모리 효율 | 모든 타일에 같은 점수 → 엔트로피 선택 = 랜덤 |
| Layout Affinity | 레이어별 최적 레이아웃 차이 | Softmax도 LINEAR도 같은 레이아웃 → 복사기 문제 |
| Cache Efficiency | SRAM 활용도와 오버플로우 | SRAM 초과 타일에 높은 점수 → 물리적으로 불가능한 선택 |

## 4. 전환 비용(Transition Cost)과 자동 가중치

### 4.1 전환 비용이란?

인접 레이어가 서로 다른 설정을 선택하면, 그 사이에 데이터 변환이 필요하다:

| 전환 종류 | 예시 | 비용 |
|-----------|------|------|
| Layout 전환 | ROW_MAJOR → COL_MAJOR | 1.0 (행렬 전치 필요) |
| Layout 전환 | ROW_MAJOR → BLOCK_TILED | 0.5 (부분 재배열) |
| 위치 전환 | SRAM → DRAM | 0.8 (메모리 계층 이동) |
| 타일 크기 전환 | 32×32 → 64×64 | 0.0~0.6 (면적 비율에 비례) |

이 비용들은 합산되어 **총 전환 페널티**가 된다 (최대 ~2.4).

### 4.2 전환 비용을 왜 반영하는가?

각 레이어를 독립적으로 최적화하면, 매 레이어 경계에서 데이터 재배치가 발생할 수 있다.
단독 최적해를 포기하더라도 전체 파이프라인의 전환 비용을 줄이는 것이 더 나을 수 있다.

예시:
```
Layer A의 단독 최적: ROW_MAJOR (score = 1.2)
Layer B의 단독 최적: COL_MAJOR (score = 1.1)
→ A→B 전환 비용: 1.0 (전치 필요)

Layer A가 COL_MAJOR를 선택하면 (score = 1.05):
→ A→B 전환 비용: 0.0
→ 총 이득: 0.15 손해 vs 1.0 절약 = 순이익 0.85
```

### 4.3 가중치(transition_weight)의 역할

전환 페널티는 무차원 수(0~2.4)이고, base score도 무차원 수(0~1.4)이다.
가중치는 이 둘의 상대적 중요도를 조절한다:

```python
adjusted_score = base_score - penalty × weight
```

- weight = 0.01 → 전환 비용을 거의 무시 → 각 레이어가 독립적으로 최적화
- weight = 0.5 → 전환 비용을 강하게 반영 → 이웃과 일관된 설정 선호

### 4.4 자동 파생 공식

```python
weight = clamp(reference_tile_bytes / sram_bytes, 0.01, 0.5)
```

| 변수 | 의미 |
|------|------|
| `reference_tile_bytes` | 32×32 float32 = 4096 bytes (중간 크기 기준 타일) |
| `sram_bytes` | 하드웨어의 SRAM 용량 |

**물리적 직관**: SRAM이 작으면 타일이 SRAM의 큰 비율을 차지한다. 이 상황에서 레이아웃 전환은 SRAM 내 데이터를 완전히 교체하는 것에 가까워, 비용이 상대적으로 크다.

| 스펙 | SRAM | weight | 해석 |
|------|------|--------|------|
| toy_gpu | 64KB | 0.0625 | 기준 타일이 SRAM의 6% → 전환 부담 작음 |
| tight_gpu | 16KB | 0.25 | 기준 타일이 SRAM의 25% → 전환이 체감됨 |
| stress_gpu | 12KB | 0.333 | 기준 타일이 SRAM의 33% → 전환 = 주요 비용 |

### 4.5 왜 수동 0.1이 아니라 자동 파생인가?

수동 값의 문제:
- toy_gpu(64KB)와 stress_gpu(12KB)에서 전환 비용의 실제 부담이 5배 차이나는데, 같은 0.1을 쓰면 stress_gpu에서 전환 비용을 과소평가
- 새 하드웨어 스펙을 추가할 때마다 적절한 weight를 수동으로 찾아야 함
- 자동 파생은 "SRAM 대비 타일 점유율"이라는 물리적 근거에서 출발하므로, 어떤 스펙이든 합리적인 값을 자동으로 산출

검증 결과:
- 기존 시나리오(toy_gpu, stress_gpu)에서 자동 파생 weight가 수동 0.1과 동일한 최종 선택을 내림 → 하위 호환
- SRAM이 극도로 작은 환경(8KB)에서만 차이 발생: 자동 파생이 전환 비용을 올바르게 중시하여 이웃과 같은 layout 선택

## 5. 제약 전파(Propagation)와 Threshold

### 5.1 전파가 하는 일

레이어 A가 `32×32 ROW_MAJOR SRAM`으로 붕괴되면, 인접 레이어 B의 후보 중 전환 페널티가 threshold를 초과하는 것들을 제거한다.

```
A = 32×32 ROW_MAJOR SRAM (확정)
B의 후보 중:
  32×32 ROW_MAJOR SRAM → penalty = 0.0 → 유지
  32×32 BLOCK_TILED SRAM → penalty = 0.5 → threshold(1.5) 미만 → 유지
  64×64 COL_MAJOR DRAM → penalty = 0.8+1.0+0.3 = 2.1 → threshold(1.5) 초과 → 제거
```

### 5.2 Threshold = 1.5의 근거

| Threshold | 제거율 | 의미 |
|-----------|--------|------|
| 0.5 | 94% | 극단적. 같은 레이아웃만 허용. 탐색 공간 소멸 |
| 1.0 | 76% | 공격적. DRAM+다른 레이아웃 대부분 제거 |
| **1.5** | **43%** | **균형점. 명백히 비효율적인 것만 제거** |
| 2.0 | 15% | 관대. 거의 전파 효과 없음 |
| 3.0 | 0% | 전파 무효. 모든 후보 유지 |

1.5를 선택한 이유:
- layout 전환(1.0) + 위치 전환(0.8) = 1.8 → 제거됨. 즉 **"레이아웃과 메모리 위치가 동시에 다른" 극단적 전환만 제거**
- layout만 다르거나(1.0), 위치만 다르거나(0.8) → 유지. 즉 **한 가지 차이는 허용**
- 제거율 43%는 CLAUDE.md 기준(>25%) 충족

## 6. Safety Check가 검증하는 것

Safety Check는 "결과가 좋아 보이는데, 진짜 좋은 건지 의심하자"는 연구 원칙에서 출발한다.

### Check #1: Hard Constraint 정확성

**의심**: "12개를 제거했다고 하는데, 정말 물리적으로 불가능한 것만 제거한 건가? 유효한 후보를 잘못 날리진 않았나?"

검증 방법: 각 후보를 수동으로 SRAM 용량과 alignment를 계산하여 함수 결과와 대조한다.

핵심 체크:
- DRAM 후보가 SRAM 제약에 걸리면 버그 (DRAM은 SRAM과 무관)
- 8×8 타일이 alignment에 걸리는 것이 과도한지 판단 (현재는 의도된 동작)

### Check #2: Cost Model 변별력

**의심**: "모든 후보에 비슷한 점수를 주면, 엔트로피 기반 선택이 사실상 랜덤과 같은 것 아닌가?"

검증 방법: 점수의 최대-최소 차이(spread)를 측정한다.

통과 기준: spread > 0.01이고, 최소 3단계 이상 구분되어야 한다.
현재 결과: spread = 1.19, 54개 후보가 완전히 구분됨.

### Check #3: 전파 강도

**의심**: "threshold=1.5가 너무 관대해서 전파가 명목상인 것 아닌가? 아니면 너무 엄격해서 유효한 후보를 날리는 건 아닌가?"

검증 방법: threshold를 0.5~3.0까지 변화시키며 제거율 비교.
통과 기준: threshold=1.5에서 제거율 > 25%.
현재 결과: 43%.

### Check #4: 백트래킹

**의심**: "백트래킹 0회 = 알고리즘이 완벽하다?"가 아니라 "백트래킹 코드가 한 번도 실행되지 않은 dead code"일 수 있다.

검증 방법: 의도적으로 모순을 유발하는 시나리오를 구성한다.
- **4a**: 양쪽이 상충하는 레이아웃을 강제 (ROW ↔ COL 갈등)
- **4b**: 해가 없는 극단적 제약 (threshold=0.5에서 ROW→COL 불가)

통과 기준: 최소 1개 시나리오에서 backtrack_count > 0.

### Check #5: 스케일링 & 복사기 문제

**의심**: "3개 레이어에서 98% 감소는 인상적이지만, 50개 레이어에서도 작동하나? 그리고 모든 레이어가 같은 답을 고르는 건 아닌가?"

검증 방법:
- **5a**: 동질적 레이어(Linear+ReLU 반복)를 3/10/50개로 스케일링
- **5b**: 이질적 레이어(MatMul+Softmax+LayerNorm)를 stress_gpu에서 실행

통과 기준: 이질적 레이어에서 `all_same_state = False`.
현재 결과: MatMul=32×32, Softmax=16×16으로 차별화 성공.

## 7. 전체 파이프라인 동작 예시

stress_gpu(12KB)에서 Attention Block 4개 레이어를 스케줄링하는 과정:

```
[초기 상태]
  QK_MatMul:  66개 후보 (11 타일 × 3 레이아웃 × 2 위치)
  Softmax:    66개 후보
  AV_MatMul:  66개 후보
  LayerNorm:  66개 후보
  총 탐색 공간: 264 states

[Step 1: Hard Constraint]
  SRAM 12KB 초과 타일 제거 + alignment 위반 제거
  → 96개 제거, 168 states 남음

[Step 2: 병목 우선 붕괴]
  QK_MatMul이 FLOPs 최대 (536M) → 먼저 붕괴
  비용 모델 점수 계산 → 32×32 ROW_MAJOR SRAM이 최고점
  → QK_MatMul 확정

[Step 3: 전파]
  QK_MatMul(32×32 ROW) → Softmax: 전환 페널티 > 1.5인 후보 20개 제거
  → Softmax 후보 22개 남음

[Step 4: 나머지 붕괴]
  Softmax: 엔트로피 계산 → 16×16 ROW_MAJOR가 최고점 (32×32는 working memory 초과)
  AV_MatMul → 32×32 ROW_MAJOR (QK_MatMul과 동일한 물리적 이유)
  LayerNorm → 32×32 ROW_MAJOR

[결과]
  264 → 4 states (98.5% 감소)
  Softmax만 다른 타일 선택 → 복사기 문제 해결
```

## 8. 현재 한계와 알려진 제약

### 알고리즘 한계
- **1D 체인만 지원**: 현재 전파는 레이어 0→1→2→3 직선만 지원. ResNet의 skip connection이나 Transformer의 multi-head 병렬 같은 DAG 구조 미지원.
- **Threshold 고정**: penalty_threshold=1.5가 모든 모델에 최적은 아님. 레이어가 100개이면 전파가 약해질 수 있음.

### 비용 모델 한계
- **실제 실행 시간과의 괴리**: Roofline 모델은 이론적 상한. 캐시 충돌, warp divergence 등은 반영하지 않음.
- **Conv2D 미성숙**: im2col 기반 memory traffic 모델이 아직 없음. 현재는 MatMul과 동일하게 취급.

### 검증 한계
- **실제 GPU 벤치마크 없음**: 비용 모델의 점수가 실제 실행 시간과 상관이 있는지 검증되지 않음.
- **가상 스펙 기반**: stress_gpu의 12KB SRAM은 현존 GPU에 없는 스펙. 실제 A100(192KB)에서는 문제가 너무 쉬울 수 있음.

## 부록: 용어 정리

| 용어 | 의미 |
|------|------|
| **Superposition (중첩)** | 레이어가 아직 결정되지 않은 상태. 모든 가능한 HWState를 동시에 보유 |
| **Collapse (붕괴)** | 레이어의 후보 중 하나를 확정하고 나머지를 폐기 |
| **Propagation (전파)** | 확정된 레이어의 제약이 이웃 레이어의 후보를 줄이는 과정 |
| **Backtracking (백트래킹)** | 붕괴 결과 모순이 발생하면 이전 상태로 되돌리는 안전장치 |
| **Entropy (엔트로피)** | 후보들의 점수 분포가 얼마나 "확실한지". 낮을수록 한 후보가 압도적 |
| **Roofline Model** | 연산량 vs 메모리 대역폭으로 성능 병목을 판단하는 분석 프레임워크 |
| **Working Memory** | 타일 하나를 처리할 때 SRAM에 동시에 올려야 하는 총 메모리 |
| **Transition Penalty** | 인접 레이어 간 설정이 달라서 발생하는 데이터 변환 비용 |
| **HWState** | 하나의 하드웨어 구현 후보. (tile_m, tile_n, layout, location)의 조합 |
